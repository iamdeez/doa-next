# Spec: 020-data-migration-cutover

> Branch: 020-data-migration-cutover | Date: 2026-07-05 | Version: v1.1.0

## 배경 및 목적

DOA Market 은 기존 AWS 기반 MSA 18개 서비스 오픈마켓을 Fly.io + NestJS 모듈러 모놀리스로 재구축해왔다(001~019). 신규 시스템은 18개 도메인 모듈 전부 실구현·테스트 green(unit 404·static 60·e2e 125/127) 상태로 기능 개발이 사실상 완료되었으나, **레거시 AWS 시스템에 있는 실 운영 데이터를 신규 시스템으로 옮기고 실제 트래픽을 전환하는 절차(데이터 이관·컷오버)가 아직 설계되어 있지 않다.** `docs/specs/` 전체(001~019)에 이관·컷오버 관련 spec 이 부재함을 확인했다.

레거시 18개 서비스는 서비스별 RDS PostgreSQL 을 사용하며 현재도 라이브로 실 트래픽을 서빙 중이다(정지 상태에서의 일괄 복사는 불가능). 신규 시스템은 동일 엔진(PostgreSQL)의 단일 인스턴스·8개 스키마 구조이므로 엔진 변환은 불필요하나, 18개 개별 서비스 데이터 모델과 8개 스키마(33테이블) 데이터 모델 간 구조가 다를 수 있다(신규 시스템은 레거시를 그대로 복제한 것이 아니라 처음부터 재설계됨).

사용자는 완전 무중단(dual-write/CDC) 대신 **짧은 유지보수 윈도우(최대 1시간)** 방식을 선택했다: 공지된 윈도우 동안 레거시 쓰기를 중단하고, 최종 델타를 이관한 뒤, 정합성을 검증하고, 트래픽을 신규 시스템으로 전환한다. 전환 방식은 8개 스키마 전체를 한 번에 옮기는 **빅뱅(big-bang) 일괄 전환**이다 — 주문(orders)·결제(payments)·정산(settlements) 3개 스키마가 강하게 연쇄되어 있어(주문 완료→결제 완료→정산) 일부만 먼저 전환하면 참조 정합성이 깨질 위험이 크기 때문이다.

본 spec 은 이 이관·컷오버를 **무엇을, 왜** 수행해야 하는지를 정의한다. 구체적인 이관 도구(pg_dump/logical replication/AWS DMS 등)·스크립트 구현·실행 환경(Fly.io 워커 vs 별도 러너) 선택은 plan.md(HOW)로 위임한다.

## 사용자 스토리

- **US-001**: 운영자로서, 최대 1시간의 유지보수 윈도우 내에 레거시 AWS 시스템의 데이터를 신규 Fly Postgres 시스템으로 이관하고 트래픽을 전환하여, 서비스 중단을 최소화하며 컷오버를 완료하고 싶다.
- **US-002**: 운영자로서, 이관된 데이터가 레거시와 정확히 일치함을 자동화된 검증으로 확인하여, 데이터 유실·오류 없이 전환되었음을 확신하고 싶다.
- **US-003**: 운영자로서, 컷오버 실행 절차를 순서화된 런북으로 문서화하여, 실행 시 실수를 방지하고 필요 시 반복(리허설) 가능하게 하고 싶다.
- **US-004**: 운영자로서, 컷오버 직후 신규 시스템에서 심각한 결함이 발견되면 레거시 시스템으로 신속히 되돌릴 수 있는 롤백 경로를 갖고 싶다.
- **US-005**: 고객·판매자로서, 컷오버 유지보수 윈도우에 대해 사전에 명확한 안내를 받아 서비스 접근 불가 시간을 예측하고 싶다.

## 기능 요구사항

- **FR-001**: 이관 절차는 신규 8개 스키마(users·products·commerce·orders·payments·settlements·admin·files) 전체에 대해 레거시 18개 서비스별 RDS PostgreSQL 로부터의 데이터 이관을 지원해야 한다.
- **FR-002**: 이관 절차는 유지보수 윈도우 개시 전에 레거시 데이터의 벌크 사전 복사(pre-copy)를 완료해야 한다.
- **FR-003**: 유지보수 윈도우 개시 시점에 레거시 18개 서비스에 대한 쓰기(write) 요청을 일관되게 차단(점검 모드 또는 DB 레벨 접근 제한)해야 한다.
- **FR-004**: 쓰기 차단 이후 사전 복사 이후 변경분(최종 델타)을 윈도우 내에 추가 이관해야 한다.
- **FR-005**: 이관 완료 직후, 레거시·신규 데이터에 대해 다음 3종의 자동화된 정합성 검증을 수행해야 한다: (a) 스키마·테이블별 레코드 수 대조, (b) 금전 필드(주문 총액·결제 금액·정산 금액) 합계 대조, (c) 무작위 샘플 레코드에 대한 체크섬/해시 비교.
- **FR-006**: FR-005 의 검증이 모두 통과한 경우에만 트래픽을 신규 시스템으로 전환해야 한다.
- **FR-007**: FR-005 의 검증 중 하나라도 실패하면 절차는 트래픽 전환을 진행하지 않고, 레거시 쓰기 차단을 해제하여 레거시 시스템에서 정상 서비스를 재개해야 한다(point of no return 이전이므로 데이터 유실 없이 되돌릴 수 있어야 한다).
- **FR-008**: 트래픽 전환 이후 신규 시스템에서 새로운 주문 또는 결제가 1건이라도 발생하면 그 시점부터 point of no return 으로 간주하고, 그 이후의 레거시 시스템으로의 자동 롤백(역이관)은 본 절차의 지원 범위에서 제외해야 한다.
- **FR-009**: 이관 절차는 레거시 18개 서비스 데이터 모델의 각 테이블·필드가 신규 8개 스키마의 어느 테이블·필드로 대응되는지를 명시하는 필드 단위 매핑 명세를 산출물로 가져야 한다.
- **FR-010**: 매핑 명세 산출 과정에서 1:1 대응이 되지 않는 항목(병합·분할·재구조화가 필요한 필드)이 발견되면, 해당 변환 규칙을 명시적으로 문서화해야 한다.
- **FR-011**: 이관 절차는 실행 전 레거시 데이터 규모(테이블별 행 수·전체 용량)를 측정하고, 유지보수 윈도우(최대 1시간) 내 완료 가능 여부를 사전 평가해야 한다.
- **FR-012**: FR-011 의 사전 평가 결과 예상 소요 시간이 윈도우의 안전마진(NFR-005)을 초과할 것으로 판단되면, 실행 전 사용자에게 재확인(윈도우 재산정 또는 부분 사전이관 전략 협의)을 요청해야 한다.
- **FR-013**: 컷오버 실행 절차는 단계별 체크포인트·담당자·각 단계의 롤백 트리거 조건을 포함하는 순서화된 런북 문서로 제공되어야 한다.
- **FR-014**: 강결합 도메인(orders·payments·settlements 스키마)은 동일 컷오버 시점에 함께 전환되어 스키마 간 참조 정합성(예: `settlement.orderItemId` ↔ `order_items`)이 깨지지 않아야 한다.
- **FR-015**: 개인정보(사용자 PII) 및 결제 관련 데이터는 이관 과정(전송 채널)에서 평문 노출 없이 암호화되어 전송되어야 한다.
- **FR-016**: 컷오버 유지보수 윈도우 일정은 최소 D-3일 전 영향받는 사용자(고객·판매자)에게 사전 공지되어야 한다.
- **FR-017**: `file_assets` 메타데이터(레코드)의 이관은 본 절차에 포함하되, 실 파일 바이너리(오브젝트) 이관은 범위 외로 한다(근거는 "범위 외" 절 참조).

## 비기능 요구사항

- **NFR-001**: 유지보수 윈도우(쓰기 차단 시작 ~ 트래픽 전환 완료)는 60분을 초과하지 않아야 한다.
- **NFR-002**: 정합성 검증의 레코드 수 대조는 레거시-신규 간 불일치 0건이어야 한다.
- **NFR-003**: 정합성 검증의 금전 필드(주문 총액·결제 금액·정산 금액) 합계 대조는 레거시-신규 간 오차 0원이어야 한다.
- **NFR-004**: 이관 데이터 전송 채널은 TLS(sslmode=require 이상)를 사용해야 한다.
- **NFR-005**: FR-005 검증 및 GO/NO-GO 판단은 윈도우 개시 후 50분 이내에 완료되어야 한다(윈도우 종료 전 최소 10분의 롤백 판단·실행 여유 확보).
- **NFR-006**: 이관 절차 실행 이력(각 단계 시작·종료 시각, 검증 결과)은 감사 가능한 형태로 기록되어야 한다.

## 수용 기준

- **SC-001** (`FR-001` 관련): 이관 파이프라인 실행 후 8개 신규 스키마의 모든 테이블에 레거시 대응 데이터가 존재함을 확인한다. [env:e2e-db]
- **SC-002** (`FR-002` 관련): 벌크 사전 복사 완료 시점에 레거시-신규 간 레코드 수 차이가 사전 정의된 델타 임계치(직전 N분 변경률 기준) 이내임을 확인한다. [env:integration]
- **SC-003** (`FR-003` 관련): 윈도우 개시 시각 이후 레거시 API 가 쓰기 요청(POST/PUT/PATCH/DELETE)에 대해 일관되게 차단 응답(예: 503 점검 모드)을 반환함을 확인한다. [env:integration]
- **SC-004** (`FR-004` 관련): 델타 이관 완료 시점에 레거시와 신규의 마지막 변경 레코드(updatedAt 등 기준)가 일치함을 확인한다. [env:integration]
- **SC-005** (`FR-005`·`NFR-002` 관련): 정합성 검증 리포트에서 모든 대상 테이블의 레코드 수가 레거시=신규로 100% 일치함을 확인한다(불일치 0건). [env:e2e-db]
- **SC-006** (`FR-005`·`NFR-003` 관련): 정합성 검증 리포트에서 orders.totalAmount·payments.amount·settlements.payoutAmount 합계가 레거시=신규로 일치함을 확인한다(오차 0원). [env:e2e-db]
- **SC-007** (`FR-005` 관련): 정합성 검증이 스키마당 최소 100건 또는 전체의 1% 중 큰 값의 무작위 샘플에 대해 체크섬/해시가 일치함을 확인한다. [env:e2e-db]
- **SC-008** (`FR-006` 관련): 런북 문서에 "SC-005~SC-007 전부 PASS 시에만 트래픽 전환(GO) 단계 진행" 이 명시되어 있음을 확인한다. [env:static]
- **SC-009** (`FR-007` 관련): 검증 실패 시나리오(리허설)를 재현했을 때, 절차가 트래픽 전환을 진행하지 않고 레거시 쓰기 차단이 해제되어 정상 서비스가 재개됨을 확인한다. [env:integration]
- **SC-010** (`FR-008` 관련): 런북 문서에 "컷오버 후 신규 주문/결제 1건 발생 시점부터 롤백 불가(point of no return)" 경고 단계가 명시되어 있음을 확인한다. [env:static]
- **SC-011** (`FR-009` 관련): 매핑 명세 문서에 18개 레거시 서비스 데이터 모델과 8개 신규 스키마 간 필드 단위 매핑표가 존재하며, 신규 스키마의 모든 테이블이 매핑표에 최소 1회 이상 등장함을 확인한다. [env:static]
- **SC-012** (`FR-010` 관련): 매핑표 내 "1:1 아님"으로 표시된 각 항목에 변환 규칙 설명이 누락 없이 기재되어 있음을 확인한다. [env:static]
- **SC-013** (`FR-011` 관련): 이관 실행 전 산출되는 사전 평가 리포트에 테이블별 행 수·예상 소요 시간·윈도우 대비 여유율이 기재되어 있음을 확인한다. [env:static]
- **SC-014** (`FR-012` 관련): 사전 평가 리포트의 예상 소요 시간이 NFR-005 안전마진(50분)을 초과하는 경우, 런북에 "진행 전 사용자 재확인 필수" 체크포인트가 명시되어 있음을 확인한다. [env:static]
- **SC-015** (`FR-013` 관련): 런북 문서의 각 단계에 담당자·체크포인트·롤백 트리거 조건이 누락 없이 기재되어 있음을 확인한다. [env:static]
- **SC-016** (`FR-014` 관련): 정합성 검증 시 orders·payments·settlements 3개 스키마 간 교차 참조 무결성이 100% 유지됨(고아 레코드 0건)을 확인한다. [env:e2e-db]
- **SC-017** (`FR-015`·`NFR-004` 관련): 이관 파이프라인의 DB 연결 설정에 TLS(sslmode=require 이상)가 적용되어 있음을 설정 검토로 확인한다. [env:static]
- **SC-018** (`FR-016` 관련): 런북 문서에 컷오버 최소 D-3일 전 사용자 공지 체크포인트(공지 채널·완료 확인란)가 포함되어 있음을 확인한다. [env:static]
- **SC-019** (`FR-017` 관련): 정합성 검증 대상에 `file_assets` 메타데이터 레코드 수 대조가 포함되고, 실 파일 바이너리 전송 검증은 스코프 문서상 명시적으로 제외되어 있음을 확인한다. [env:static]
- **SC-020** (`NFR-001` 관련): 컷오버 리허설(dry-run) 또는 실제 실행 로그에서 "쓰기 차단 시작" ~ "트래픽 전환 완료" 시각 차이가 60분 이하임을 확인한다. [env:e2e-db]
- **SC-021** (`NFR-005` 관련): 리허설 로그에서 FR-005 검증 및 GO/NO-GO 판단이 윈도우 개시 후 50분 이내에 완료됨을 확인한다. [env:e2e-db]
- **SC-022** (`NFR-006` 관련): 이관 절차 실행 로그(각 단계 시작·종료 시각, 검증 결과)가 감사 가능한 형태(파일 또는 감사 테이블)로 저장됨을 확인한다. [env:integration]

## 범위 외

- **레거시 AWS 인프라 해체(decommission)**: 컷오버(트래픽 전환) 완료 이후 레거시 AWS 리소스(RDS, EC2/ECS 등)를 실제로 종료·삭제하는 절차는 본 spec 범위 외다 — 별도 후속 spec 으로 진행한다(사용자 확정).
- **실 파일 바이너리(오브젝트) 이관(S3 → R2)**: `apps/backend/src/modules/file/file.module.ts` 확인 결과, 신규 시스템의 `FILE_STORAGE` DI 는 아직 `StubFileStorage`(무네트워크 stub)로 바인딩되어 있어 실 Cloudflare R2 연동 자체가 존재하지 않는다(`context.md` 기 기재 사항 — "실 R2 연동은 후속"). 실 파일 이관은 R2 실연동(Port 구현체 교체) 완료가 선행 조건이므로 별도 spec 으로 분리한다. 본 spec 은 `file_assets` 메타데이터(레코드)만 이관 대상에 포함한다(FR-017/SC-019, ASM-012).
- **완전 무중단(dual-write/CDC) 아키텍처**: 사용자가 짧은 유지보수 윈도우(최대 1시간) 방식을 확정 선택함에 따라, 이중 쓰기·Change Data Capture 기반의 진짜 zero-downtime 아키텍처는 범위 외다.
- **point of no return 이후의 자동 역이관**: 컷오버 후 신규 시스템에서 새 주문/결제가 발생한 시점 이후, 신규 데이터를 유실 없이 레거시로 되돌리는 자동화된 역이관 기능은 지원하지 않는다(FR-008).
- **결제 PG·R2 등 외부 연동 자체의 완성**: 결제 게이트웨이는 현재 stub(`PaymentGatewayPort` + stub) 상태이며 실 PG 연동 완성은 독립적인 별도 spec 이다. 본 spec 은 데이터(payments 스키마 레코드) 이관만 다루며, 실 PG 연동 완성 여부와 무관하게 진행 가능하다.
- **Fly Postgres HA 구성 여부 결정**: `infra.md §8` 이 "Fly Postgres 단일 장애점 — HA 옵션 또는 자동 백업+PITR 완화 필요, 로드맵 6단계(컷오버) 이전 결정" 으로 이미 플래그해 두었다. 본 spec 은 이 인프라 결정 자체(HA 도입 여부)를 강제하지 않으며, Planning 단계에서 컷오버 시점의 가용성 요구 수준에 따라 별도로 검토·결정한다.
- **사후 검증 활동 실행 방식**: 실제 레거시 AWS RDS 접속이 필요한 검증(SC-001~SC-007, SC-016, SC-020, SC-021)은 파이프라인 내에서 자동 실행할 수 없다(레거시 자격증명·네트워크 접근이 사용자 환경에 있음). 아래 옵션 중 **옵션 A(사용자 실행 + 결과 전달)** 를 기본으로 채택한다 — 필요 시 후속 단계(Deploy/Test Agent)에서 재확인한다.
  - 옵션 A (채택): main session/Planning 산출물이 실행 절차(스크립트·명령)를 제시 → 사용자가 실제 레거시·신규 환경에서 실행 → 결과(리포트)를 전달 → Test/Deploy Agent 가 결과를 검증.
  - 옵션 B: 사용자가 직접 전체 리허설을 구축·실행·결과 전달.
  - 옵션 C: 실제 리허설 스킵, 정적 검증(런북·매핑 명세 문서 완결성)만으로 마감 — 본 spec 에서는 채택하지 않음(실 데이터 이관이라는 spec 의 본질상 정적 검증만으로는 SC-001~007 등 핵심 수용 기준을 충족할 수 없음).

### 사후 운영 검증 피드백 사이클

본 spec 파이프라인 종료(런북·매핑 명세 확정) 후에도, 실제 컷오버 실행은 사용자 환경(레거시 AWS 접근)에서 이루어지므로 아래 사후 발견 가능 시나리오를 대비한다.

1. 파이프라인 종료 후 사용자가 실제 환경에서 점검할 가능성이 있는 시나리오: (a) 실측 데이터 규모가 예상보다 커서 60분 윈도우 초과, (b) 매핑 명세에 없던 레거시 필드/테이블 발견(리허설 중), (c) 정합성 검증 중 예상치 못한 불일치(예: 레거시 타 서비스의 숨은 쓰기 경로), (d) 실제 트래픽 전환(DNS/LB) 전환 지연.
2. 위 시나리오로 결함이 발견되면: 결함 내용을 본 spec.md "미결 사항" 또는 별도 patch spec 입력으로 사용 → main session 의 "spec 수정" 이벤트 → 1단계 재진입(cycle N+1) 또는 별도 patch spec 진입.
3. 사후 검증 결과는 컷오버 실행 완료 후 `docs/specs/v1.1.0/CHANGES.md` 및 `context.md` 갱신을 통해 추적한다(사용자와 별도 협의 필요 시 그 시점에 진행).

## 미결 사항

없음 — 팀리드가 전달한 사용자 확정 사항(다운타임 허용도·컷오버 방식·범위·롤백 요구) 및 미답변 항목에 대한 합리적 가정(assumptions.md ASM-001~012)으로 모든 항목이 해소되었다. 설계를 물질적으로 바꿀 수 있는 두 가지 리스크(데이터 규모 미상 — ASM-001, 매핑 복잡도 미상 — ASM-002)는 Planning/Design 단계에서 실측·매핑 명세 산출로 조기 확인하도록 FR-011/FR-012/FR-009/FR-010 에 명문화했다.
