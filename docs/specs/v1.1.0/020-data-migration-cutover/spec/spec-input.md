## 카테고리별 수집 내용

### [카테고리 1] 배경 및 목적

**Q1.** 본 spec 의 "데이터 이관"이 다루는 범위는 무엇인가? (a) 레거시 AWS RDS/DynamoDB 등에 있는 실 운영 데이터를 신규 PostgreSQL 8스키마로 옮기는 것만인가, (b) 트래픽 전환(DNS/로드밸런서 스위칭)까지 포함하는 "컷오버" 전체인가, (c) Cloudflare R2로의 파일(S3) 이관도 포함하는가?

**Q2.** 레거시 AWS MSA 18개 서비스가 **현재도 라이브로 프로덕션 트래픽을 서빙 중**인가? 아니면 이미 운영 중단되었거나, 처음부터 실사용자 트래픽이 없던 개발/검증 환경인가? (live 이관 vs 정지 상태 일괄 이관 여부를 좌우하는 핵심 질문)

**Q3.** 컷오버 완료의 성공 판단 기준은 무엇인가? (예: 신규 시스템 100% 트래픽 전환 + N일간 무장애 관찰, 레거시 인프라 완전 폐기 시점, 특정 KPI 등)

### [카테고리 2] 사용자 & 이해관계자

**Q4.** 컷오버 작업을 실행·모니터링하는 주체는 누구인가? (본인 단독 운영인가, 별도 운영팀이 있는가?)

**Q5.** 컷오버 중 영향을 받는 사용자 유형은 누구인가? (고객 Flutter 앱 최종 사용자, 판매자 콘솔 사용자, 관리자 — 각각 다운타임 허용도가 다를 수 있음)

**Q6.** 컷오버 일정·다운타임에 대해 사전 공지가 필요한 이해관계자(예: 입점 판매자, 결제 PG사)가 있는가?

### [카테고리 3] 핵심 기능 (정상 흐름)

**Q7.** 컷오버 방식 선호 — 위 "질문 분석 근거" Q7 참조. (a) 빅뱅(전체 일괄 전환) (b) 도메인/스키마별 점진 전환 (c) 하이브리드(약결합 도메인 선행 + 주문/결제/정산 묶음 후행) 중 어느 방향을 원하는가, 또는 별도 의견이 있는가?

**Q8.** 이관 대상 범위 우선순위 — 18개 도메인(또는 8개 신규 스키마) 전체를 한 번에 이관하는가, 아니면 우선순위(예: 사용자·상품 먼저, 정산은 나중)가 있는가?

**Q9.** Out of Scope 확인 — 본 spec 이 다루지 않을 항목은? 예: (a) 실제 이관 스크립트 구현·실행 자체 (b) 신규 PG(결제)·R2 실연동 완료 여부와 무관하게 데이터만 이관 (c) 컷오버 이후 레거시 AWS 인프라 해체(decommission) 절차. 이 중 어디까지가 본 spec 범위인지 확정 필요.

### [카테고리 4] 데이터 & 입출력

**Q10.** 레거시 AWS 데이터 소스 — 18개 MSA 서비스의 데이터가 실제로 어디에 있는가? (서비스별 개별 RDS/Aurora? 공유 RDS? DynamoDB 테이블? S3? 각 서비스가 어떤 저장소를 쓰는지 서비스별로 나열 가능한가?)

**Q11.** 데이터 규모 — 서비스별 대략적인 테이블 행 수(order, product, user 등 핵심 테이블 기준) 또는 전체 데이터 볼륨(GB)은 어느 정도인가? (이관 소요 시간·전략 결정에 필요)

**Q12.** 데이터 매핑 복잡도 — 레거시 18서비스의 데이터 모델과 신규 8스키마(users/products/commerce/orders/payments/settlements/admin/files) 매핑이 필드 단위로 1:1에 가까운가, 아니면 스키마 재구조화(예: 여러 레거시 테이블을 병합, 정규화 방식 변경)가 필요한 수준인가? (목록형 응답 요약·미리보기 필드 관련 질문(Q12-1)은 본 spec 성격상 해당 없음으로 처리)

### [카테고리 5] 제약조건 (수치로 표현)

**Q13.** 다운타임 허용도 — 위 "질문 분석 근거" Q13 참조. 완전 무중단(진짜 zero-downtime, dual-write/CDC 필요)을 요구하는가, 아니면 유지보수 윈도우(구체적으로 몇 분~몇 시간까지 허용 가능한가?)를 허용하는가?

**Q14.** 컷오버 목표 일정 — 언제까지 완료되어야 하는가? (구체적 날짜 또는 "N주 이내" 등)

**Q15.** 이관 처리량/시간 요구사항 — 이관 작업 자체가 완료되어야 하는 시간 제약이 있는가? (예: "유지보수 윈도우 N시간 내에 전체 이관 완료" 등 수치로)

**Q16.** 규제/법적 요구사항 — 개인정보(사용자 PII)·결제 정보 이관 시 준수해야 할 규정(마스킹, 암호화, 접근 로그 등)이 있는가?

### [카테고리 6] 운영 환경

**Q17.** 이관 작업 실행 환경 — 로컬 PC에서 스크립트 실행? Fly.io 상의 별도 잡/워커로 실행? 별도 이관 전용 서버/컨테이너를 프로비저닝하는가? (infra.md §8 "Fly Postgres 단일 장애점" 항목이 "로드맵 6단계(컷오버) 이전 결정" 필요 사항으로 이미 명시되어 있어, 본 spec 에서 HA 여부 결정이 필요한지도 함께 확인 요망)

**Q18.** 레거시 AWS 접근 권한 — 이관 작업 시점에 AWS 자격증명(RDS 접근, DynamoDB 접근, S3 접근 등)이 여전히 유효한가, 아니면 이미 만료/폐기되었는가?

**Q19.** 배포·운영 담당 — 컷오버 실행 승인 및 실시간 모니터링은 누가 담당하는가? (Q4 와 동일 인물/팀인지, 별도 승인권자가 있는지)

### [카테고리 7] 예외 & 실패 시나리오

**Q20.** 롤백 요구 — 컷오버 실패(예: 이관 후 신규 시스템에서 심각한 결함 발견) 시 레거시 AWS 시스템으로 되돌리는 롤백 경로가 필수인가?

- Q1/Q9 (범위): 이관 설계 + 실제 실행 + 정합성 검증 + 컷오버 런북까지 포함, 레거시 AWS 해체(decommission)는 범위 외 — spec.md "범위 외" 절에 명시.
- Q20/Q21 (롤백): 필수, point of no return = 컷오버 후 신규 주문/결제 발생 시점 — FR-007·FR-008·SC-009·SC-010 에 반영.
