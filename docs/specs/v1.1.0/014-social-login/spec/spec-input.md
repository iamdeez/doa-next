## 카테고리별 수집 내용

### [카테고리 1] 배경 및 목적

- **Q1 (왜 만드는가)**: 소비자 앱 소셜 로그인 지원으로 가입 장벽 낮춤. 013-flutter-customer-phase2에서 OAuth 인프라 규모로 014로 분리.
- **Q2 (현재 한계)**: LoginScreen에 카카오·구글·네이버 소셜 버튼 3종이 시각적으로만 존재 (`_SocialRow` 위젯, onTap 핸들러 없음 — login_screen.dart 직접 확인). 백엔드에 소셜 인증 인프라 없음.
- **Q3 (성공 기준)**: 카카오·구글·네이버 각각 소셜 로그인 → JWT(accessToken·refreshToken) 발급 성공, 기존 이메일 계정과 자동 연동 동작, 별도 회원가입 단계 없이 소셜 로그인만으로 가입 완료.

### [카테고리 2] 사용자 & 이해관계자

- **Q4 (사용 대상)**: 소비자 앱(Flutter) 사용자 — 기존 이메일 계정 보유자 및 신규 사용자 모두 대상.
- **Q5 (기술 수준)**: 소비자 (모든 기술 수준, 버튼 탭만으로 완료되어야 함).
- **Q6 (이해관계자)**: 소비자(직접 사용자), 백엔드 개발팀(auth 모듈 확장).

### [카테고리 3] 핵심 기능

- **Q7 (필수 기능)** (우선순위 순):
  1. 카카오 소셜 로그인 (백엔드 검증 + JWT 발급)
  2. 구글 소셜 로그인 (백엔드 검증 + JWT 발급)
  3. 네이버 소셜 로그인 (백엔드 검증 + JWT 발급)
  4. 자동 연동 (소셜 이메일 = 기존 계정 이메일 → 기존 계정 연결)
  5. 소셜 전용 신규 가입 (기존 계정 없음 → 신규 생성, password null)
  6. LoginScreen 소셜 버튼 탭 핸들러 활성화 (현재 미구현 → 구현)
- **Q8 (있으면 좋지만 필수 아닌)**: 없음 (명시 결정)
- **Q9 (명시적 제외)**: 소셜 계정 연동 해제(unlink), 소셜 계정 목록 조회, 프로필 이미지 저장·표시, 실 OAuth Provider 앱 등록·크레덴셜 발급, 실 Provider 네트워크 호출 E2E 테스트, 로그인 상태에서 소셜 계정 추가 연동.

### [카테고리 4] 데이터 & 입출력

- **Q10 (데이터 모델)**: 별도 `social_accounts` 테이블 — users 스키마에 속하며 users 테이블과 1:N 관계. 한 사용자가 복수 소셜 제공자 연동 가능.
- **Q11 (저장 데이터 범위)**: 표준 범위 — `provider`(카카오·구글·네이버 식별자), `providerId`(제공자 내부 고유 ID), `email`, `name`. 프로필 이미지 미포함.
- **Q12 (민감도·추가정보)**: 소셜 크레덴셜(Client ID·Secret)은 환경변수 관리, 미발급 상태 — 환경변수 명세만 spec에 기록, 실 값 발급은 운영 셋업 단계. 추가 정보 수집(프로필, 전화번호 등) 없음.

### [카테고리 5] 제약조건

- **Q13 (기술 스택)**: Flutter (소비자 앱 기존 스택), NestJS (백엔드 기존 스택) — SDK·Passport 전략 선택은 Planning 위임.
- **Q14 (일정)**: 명시적 데드라인 없음.
- **Q15 (성능)**: 소셜 로그인 백엔드 API P95 3초 이내 (외부 제공자 사용자 정보 조회 포함). Constitution 기본 API P95 1초는 외부 제공자 호출로 인해 본 spec에서 3초로 완화 명시.
- **Q16 (보안·법규)**: 각 OAuth 제공자 Client ID·Secret은 환경변수로 관리, 개발/운영 환경별 독립 값. 자동 연동 시 제공자 이메일 소유권 검증은 해당 OAuth 제공자를 신뢰(ASM-001).

### [카테고리 6] 운영 환경

- **Q17 (실행 환경)**: Fly.io — NestJS 백엔드(production), Flutter 앱(앱스토어 배포). 실 OAuth 앱 등록(각 제공자 개발자 콘솔)은 운영 셋업 단계 deferred.
- **Q18 (환경 분리 전제)**: 개발/운영 환경별 OAuth 앱 분리 전제 (개발자 콘솔에서 별도 앱 생성). 구체 값은 Planning 위임.
- **Q19 (배포 담당)**: 기존 011-cors 기반 배포 구조 동일.

- **배포 환경 cross-reference 결과**: 소셜 로그인은 외부 OAuth 제공자 서버 호출(카카오·구글·네이버 API). infra.md 확인 결과: Fly.io 아웃바운드 트래픽 제한 없음, NAT·proxy 이슈 해당 없음. 환경변수 관리는 `fly secrets set` 패턴 기존과 동일. 별도 인프라 제약 없음.

### [카테고리 7] 예외 & 실패 시나리오

- **Q20 (실패 처리)**: 소셜 로그인 실패(제공자 오류·네트워크 오류·이메일 미반환) 시 오류 메시지를 표시하고 이메일 로그인으로 안내.
- **Q21 (엣지 케이스)**: (1) 이메일 null → 로그인 거부 / (2) 유효하지 않은 토큰 → 4xx 오류 / (3) 사용자 취소 → 로그인 화면 복귀, 오류 메시지 없음 / (4) 소셜 이메일 = 기존 계정 이메일 → 자동 연동 / (5) 이미 연동된 소셜 계정(provider+providerId 동일) → 재로그인.
- **Q22 (백업·복구)**: 해당 없음 (소셜 인증은 외부 제공자 의존, 오류 시 이메일 로그인 폴백).

## 보완 내용

- **성능 수치 명확화**: Constitution P-007 API 응답 P95 1초 기준이 있으나, 소셜 로그인 API는 외부 제공자 사용자 정보 조회를 포함하므로 본 spec에서 3초로 별도 명시.
- **자동 연동 범위 명확화**: "기존 email+password 계정"뿐 아니라 "기존 소셜 전용 계정"(다른 provider)과의 이메일 일치 시에도 자동 연동. users.email @unique 제약으로 인해 중복 계정 방지를 위해 범위 확장.
- **실 e2e 전제**: 실 OAuth 크레덴셜 미발급 상태이므로 SC 검증은 stub/mock 사용, 실 Provider 네트워크 호출 없음.
