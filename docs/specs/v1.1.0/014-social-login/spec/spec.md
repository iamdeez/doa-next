## 배경 및 목적

DOA Market 소비자 앱(Flutter)에서 카카오·구글 소셜 로그인을 제공한다.

> **[정합화 노트 — Naver 제외 (2026-07-02, GAP-014-09)]** 본 spec 은 최초 카카오·구글·네이버 3제공자로 설계되었으나, 구현·검증 과정에서 **Naver 는 SEC-001(토큰 앱바인딩 검증 수단 부재로 인한 계정 탈취)** 근거로 **이번 릴리즈에서 완전 제외**되었다. 아래 문서의 네이버 관련 US-003·FR-012·SC-009·SC-013 및 FR-001/NFR-004/SC-018 의 네이버 부분은 **범위 외(이월)** 로 간주한다. **실제 활성 제공자: 카카오·구글 2종.** Naver 는 서버 authorization code + client_secret 교환 방식 검증을 갖춘 별도 spec 으로 재도입한다.

**현재 상태** [코드현황 확인 — login_screen.dart 직접 read 완료]:
- `mobile/customer_app/lib/features/auth/login_screen.dart`의 `_SocialRow` 위젯에 카카오·구글·네이버 소셜 버튼 3종이 시각적으로만 존재한다. `GestureDetector` 또는 `onTap` 핸들러가 없는 순수 `Container` 위젯으로, 탭 시 아무 동작이 없다.
- 백엔드 auth 모듈에 소셜 인증 인프라(소셜 토큰 검증, social_accounts 테이블)가 없다.

**이월 경과**:
- v1.1.0/009: 소셜 버튼 시각적 플레이스홀더 구현, 기능 구현은 별도 spec으로 이월.
- v1.1.0/013: OAuth 인프라 규모로 014로 재이월 결정.

**목적**: 소비자가 별도 회원가입 없이 소셜 계정으로 즉시 DOA Market을 이용할 수 있게 함으로써 가입 장벽을 낮춘다.

## 기능 요구사항

### 백엔드 — 소셜 로그인 API

**FR-001**: 백엔드는 소셜 제공자 식별자(카카오·구글 중 하나 — 네이버는 이번 릴리즈 제외, SEC-001)와 클라이언트가 획득한 인증 토큰을 요청 바디로 수신하여 사용자를 인증하는 소셜 로그인 API를 제공한다.

**FR-002**: 백엔드는 수신한 토큰을 해당 소셜 제공자 서버에 검증하여 providerId·email·name을 획득한다.

**FR-003**: 소셜 제공자로부터 이메일이 반환되지 않는 경우 로그인을 거부하고 적절한 오류 응답을 반환한다.

### 백엔드 — 계정 처리 (우선순위 순)

**FR-004**: 요청한 소셜 계정(provider + providerId)과 이미 연동된 사용자 계정이 존재하는 경우, 해당 사용자로 로그인한다(기존 소셜 계정 재로그인).

**FR-005**: FR-004에 해당하지 않고, 소셜 제공자로부터 획득한 이메일이 기존 사용자 계정(이메일+비밀번호 계정 또는 기존 소셜 전용 계정)의 이메일과 동일한 경우, 해당 소셜 계정을 기존 계정에 연동하고 기존 계정으로 로그인한다(자동 연동). users.email 고유 제약을 보존하기 위해 새 계정 생성 없이 기존 계정에 연결한다.

**FR-006**: FR-004·FR-005에 해당하지 않는 경우, 소셜 제공자로부터 획득한 email·name으로 신규 사용자 계정을 생성하고 해당 소셜 계정을 연동하여 로그인한다(소셜 전용 신규 가입).

**FR-007**: 소셜 전용 신규 가입 사용자(FR-006)의 password는 null로 저장된다.

### 백엔드 — JWT 발급

**FR-008**: 소셜 로그인 성공(FR-004·FR-005·FR-006 모든 경로)시 기존 이메일 로그인과 동일한 형식의 JWT(accessToken·refreshToken)를 발급하여 반환한다.

### 백엔드 — 데이터 저장

**FR-009**: 소셜 계정 정보(provider, providerId, email, name)는 별도의 `social_accounts` 테이블에 저장한다. `social_accounts` 테이블은 `users` 스키마에 속하며 `users` 테이블과 1:N 관계(한 사용자가 복수 소셜 제공자 연동 가능)를 갖는다.

### Flutter — 소셜 버튼 활성화

**FR-010**: `LoginScreen`의 카카오 소셜 버튼 탭 시 카카오 인증 흐름이 시작된다. 인증 성공 후 획득한 토큰을 백엔드 소셜 로그인 API에 전달한다.

**FR-011**: `LoginScreen`의 구글 소셜 버튼 탭 시 구글 인증 흐름이 시작된다. 인증 성공 후 획득한 토큰을 백엔드 소셜 로그인 API에 전달한다.

**FR-012**: `LoginScreen`의 네이버 소셜 버튼 탭 시 네이버 인증 흐름이 시작된다. 인증 성공 후 획득한 토큰을 백엔드 소셜 로그인 API에 전달한다.

**FR-013**: 백엔드 소셜 로그인 API로부터 JWT 응답을 수신하면 `FlutterSecureStorage`에 저장하고 메인 화면으로 전환한다.

**FR-014**: 소셜 인증 흐름에서 사용자가 인증을 취소한 경우, 로그인 화면으로 복귀하며 오류 메시지를 표시하지 않는다.

**FR-015**: 소셜 로그인 실패(제공자 오류·네트워크 오류·이메일 미반환 등) 시 오류 메시지를 화면에 표시하고 이메일 로그인으로 안내한다.

## 비기능 요구사항

**NFR-001**: 소셜 로그인 백엔드 API 응답 P95 3초 이내(외부 제공자 사용자 정보 조회 네트워크 포함). Constitution P-007 API 일반 기준(P95 1초)은 외부 소셜 제공자 서버 호출을 포함하는 본 API에 대해 3초로 한정 완화한다.

**NFR-002**: 자동 연동(FR-005) 시 소셜 제공자가 반환하는 이메일의 소유권 검증은 해당 OAuth 제공자가 보장하는 것으로 신뢰한다(제공자 이메일 검증 신뢰 전제 — ASM-001). 추가 이메일 소유권 검증 단계를 두지 않는다.

**NFR-003**: 소셜 전용 신규 가입 사용자(FR-007, password null)가 이메일+비밀번호 로그인을 시도하는 경우 적절한 오류 응답을 반환한다.

**NFR-004**: 각 OAuth 제공자(카카오·구글 — 네이버는 이번 릴리즈 제외)의 인증 크레덴셜(Client ID·Secret 등)은 환경변수로 관리하며 개발/운영 환경별 독립 값을 지원한다. 크레덴셜 실 값 발급은 운영 셋업 단계에서 수행하며, 본 spec 검증 범위에서는 stub/mock을 사용한다.

**NFR-005**: `flutter analyze` 명령 실행 결과 0 issues (v1.1.0/009·013 기준 유지).

## 범위 외

- **소셜 계정 연동 해제(unlink)**: 연동된 소셜 계정 제거 기능 — 범위 외, 후속 spec으로 이월.
- **소셜 계정 목록 조회**: 연동된 소셜 계정 목록 API 및 UI — 범위 외.
- **프로필 이미지**: 소셜 제공자로부터 프로필 이미지 URL 수신·저장·표시 — 범위 외.
- **Naver 소셜 로그인 (완전 제외)**: SEC-001(GAP-014-08/GAP-014-10 — 네이버 공개 API 에 토큰 앱바인딩 검증 수단이 없어 클라이언트 토큰 흐름에서 계정 탈취 취약) 근거로 이번 릴리즈에서 완전 제외. 향후 **서버 authorization code + client_secret 교환 방식** 검증을 갖춘 별도 spec 으로 재도입(plan.md ADR-001 재검토). US-003·FR-012·SC-009·SC-013 및 FR-001/NFR-004/SC-018 의 네이버 부분이 이에 해당.
- **실 OAuth 제공자 앱 등록·크레덴셜 발급**: 카카오·구글 개발자 콘솔 앱 등록 및 Client ID/Secret 발급 — 운영 셋업 단계 deferred. 본 spec 검증은 stub/mock 사용.
- **실 제공자 네트워크 호출 E2E 테스트** (SC-019 포함): 실 OAuth 크레덴셜로 실제 소셜 로그인 흐름 E2E 검증 — deferred. 크레덴셜 발급 후 별도 운영 검증으로 수행.
  - 옵션 B (사용자 직접 검증): 크레덴셜 발급 후 실제 기기에서 각 제공자별 소셜 로그인 1회 테스트.
- **로그인 상태에서 소셜 계정 추가 연동**: 이미 로그인된 사용자가 추가 소셜 제공자를 연동하는 기능 — 범위 외.
- **소셜 전용 사용자 비밀번호 재설정**: 013에서 구현된 OTP 기반 비밀번호 재설정은 password=null 사용자에게 의미 없으나, 해당 흐름 수정은 범위 외.

### 사후 운영 검증 피드백 사이클 (PROC-014)

본 spec 파이프라인 종료 후 실 크레덴셜 발급 및 운영 환경 배포 시 아래 시나리오를 수동 점검한다.

1. **카카오**: 앱에서 카카오 버튼 탭 → 카카오 앱/웹 인증 → DOA Market 복귀 → 메인 화면 이동 확인.
2. **구글**: 앱에서 구글 버튼 탭 → 구글 계정 선택 → DOA Market 복귀 → 메인 화면 이동 확인.
3. **네이버**: 앱에서 네이버 버튼 탭 → 네이버 로그인 → DOA Market 복귀 → 메인 화면 이동 확인.
4. **자동 연동**: 기존 이메일 계정과 동일 이메일의 소셜 계정 로그인 시 기존 계정으로 로그인되는지 확인.
5. **기존 이메일 로그인**: 소셜 로그인 후 동일 이메일로 이메일+비밀번호 로그인이 여전히 작동하는지 확인 (자동 연동 계정).

사후 운영 검증에서 결함 발견 시: 결함 정보를 hotfix spec 입력으로 사용 → main session의 "spec 수정" 이벤트 → 별도 patch spec 진입. 미수행 시 크레덴셜 발급 완료 시점에 위 시나리오 점검 일정을 별도 수립한다.
