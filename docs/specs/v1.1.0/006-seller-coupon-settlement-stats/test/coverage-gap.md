## e2e 자동화 부재 (상세)

**현상**: 006 은 판매자 통계·정산·쿠폰 UI 화면 3개를 추가하나, 화면 흐름(통계 렌더 / 정산 표 / 쿠폰 목록 →
생성 다이얼로그 → 발급 다이얼로그)에 대한 e2e/단위 테스트가 없다. 검증은 정적 코드 리뷰 + console 빌드/
타입체크 + grep(하드코딩 0)으로 갈음했다.

**근본 원인 (UI 화면 성격 + 검증 갈음)**:
- 본 차수는 화면 추가 + 토큰 통일로, 라우트 컴파일·타입 안전성을 console typecheck/build 로, 하드코딩 0 을
  grep 으로 확인했다. 사용자 상호작용 흐름(쿠폰 생성 검증→제출→invalidate, 발급→성공 문구)은 e2e 로 회귀
  방지하는 것이 이상적이나 본 차수에 포함하지 않았다(004·005 와 동일 패턴).

**위험도**: 낮음. 화면 로직이 명료(useQuery·useMutation·클라이언트 `validate`·상태 분기)하고 console 빌드/
타입체크가 라우트 컴파일·타입 회귀를, grep 이 하드코딩 잔존을 포착한다. 다만 상호작용 흐름의 미묘한 회귀
(invalidate 누락·검증 분기 경로)는 e2e 없이는 탐지가 어렵다.

**권장 수정 방향**: 후속에 Playwright 등으로 (1) 판매 통계 카드 렌더, (2) 정산 표·상태 Badge, (3) 쿠폰 생성
검증(할인값 0/비율 100 초과 차단)→제출→목록 갱신, (4) 쿠폰 발급→성공 문구 e2e 를 추가한다(GAP-006-01).

## 쿠폰 후속 미구현 (상세)

**현상**: 쿠폰 화면에 세 가지 후속 항목이 미구현이다.
- (1) **cursor '더보기'**: `api.coupon.listSeller` facade 는 `cursor`·`take` 를 받아 `CursorPage<Coupon>` 를
  반환하나, 화면은 `listSeller()`를 인자 없이 호출하여 **첫 페이지(`data.items`)만** 렌더한다. 추가 로드
  버튼·무한 스크롤이 없다.
- (2) **발급 후 목록 즉시 반영**: `IssueCouponDialog` 의 `issue.onSuccess` 는 다이얼로그를 닫고 성공 문구만
  표시하며 `['seller','coupons']` 를 invalidate 하지 않는다. 목록의 발급/총량(`issuedCount`) 컬럼은 발급
  직후 즉시 갱신되지 않고 다음 조회 시 반영된다.
- (3) **서버 검증 에러 필드 매핑**: 생성/발급 다이얼로그는 `ApiError` 메시지를 단일 `ErrorText` 로 표시하며,
  필드별(할인값·만료일·대상 사용자) 인라인 서버 에러 매핑은 없다.

**근본 원인 (Phase 2 최소 구현 — 범위 분리)**:
- Phase 2 는 판매자 운영 화면 제공이 목표이며, 페이지네이션·낙관적 UX·서버 에러 정교화는 후속으로 분리했다
  (spec §범위 외). 현재 구현도 클라이언트 검증(010 정합)·처리 중 비활성화·에러 표시로 기본 안전성을 확보한다.

**위험도**: 낮음(의도된 범위 분리). 쿠폰 검증은 백엔드 010 서버 검증이 강제하며, 생성 mutation 은 목록을
invalidate 하여 정합성을 유지한다. 발급 결과 미즉시반영은 발급/총량 컬럼의 일시적 stale 일 뿐 데이터 손상이
아니다(다음 조회 시 정확).

**권장 수정 방향**: 후속에서 (1) cursor 기반 '더보기' 추가 로드, (2) `issueSeller` `onSuccess` 에
`invalidateQueries(['seller','coupons'])` 추가, (3) 생성/발급 다이얼로그에 필드별 서버 에러 표면 강화 +
낙관적 업데이트(`onMutate`)를 적용하고 각각의 검증 테스트를 추가한다(GAP-006-01).

## 응답 view 타입 한시 (상세)

**현상**: 통계·정산·쿠폰 응답은 백엔드가 Prisma 엔티티를 반환하고 OpenAPI 응답 content 가 미주석이다
(87 ops 중 typed 2xx content 36건 — 001 GAP-001-01 연속). 따라서 003 타입드 client 대신 `@doa/shared-types`
전이형 view 타입(금전 string) 8종을 한시 정의했다(004 와 동일 패턴).

**근본 원인 (백엔드 응답 스키마 미정의 — 점진 보강)**:
- 백엔드 컨트롤러가 응답 DTO + `@ApiResponse({ type })` 를 부여하지 않아 응답이 타입 미주석이다. 프론트는
  생성 타입에서 응답 타입을 얻지 못해 수기 view 타입으로 한시 표현한다.

**위험도**: 낮음(점진 보강 대상). view 타입은 공유 패키지(`shared-types`)에 단일 정의되어, 백엔드 응답 DTO
보강 후 한 곳에서 생성 타입(`Schemas['...']`)으로 교체 가능하다.

**권장 수정 방향**: 백엔드에 통계·정산·쿠폰 도메인별 응답 DTO + `@ApiResponse({ type })` 를 보강하고 코드젠
재생성(`openapi:gen` → `gen`)하면 view 타입을 생성 타입으로 대체할 수 있다(GAP-006-01 / 004 GAP-004-01 /
001 GAP-001-01 연속). 금전 필드는 Decimal→문자열이므로 대체 후에도 `string` 유지를 확인한다(P-005).
