## 카테고리별 수집 내용

### [카테고리 1] 배경 및 목적
- Q1: 015-naver-code-exchange 6단계 Security 재감사(security-report.md v1.1)가 SEC-015-02(Medium)·SEC-015-03(Low)을 권고사항으로 잔존시켰고, "운영 셋업 착수 전 필수 선행 조건"으로 명시했다. 본 spec 은 이 두 항목의 백엔드측 하드닝을 다룬다.
- Q2: 현재 `naver.provider.ts` 는 클라이언트가 보낸 state 를 검증 없이 네이버로 forward 만 하고, redirect_uri 는 토큰 교환 요청에 아예 포함하지 않는다.
- Q3(성공 판단 기준, 확정): (1) SEC-015-02 가 Security Agent 재감사에서 RESOLVED 로 판정, (2) SEC-015-03 이 RESOLVED 또는 잔존-권고(운영 확인 대기)로 판정, (3) state 미검증·불일치·만료 요청이 거부되는 것이 검증 가능한 SC 로 존재.

### [카테고리 2] 사용자 & 이해관계자
- Q4~Q6 (확정): 015 와 동일 — 소비자(네이버 로그인 사용자), 백엔드/모바일 개발자, 운영 담당자. 추가 이해관계자 없음.

### [카테고리 3] 핵심 기능
- Q7 (필수 기능, 우선순위순): (1) 서버측 state 발급, (2) code-exchange 시 state 검증(일치·미만료·1회성), (3) 검증 실패 시 거부, (4) 카카오·구글은 state 검증 대상에서 제외(회귀 방지), (5) redirect_uri 환경변수 조건부 지원.
- Q8 (있으면 좋으나 필수 아님): 없음(본 spec 은 좁은 하드닝 patch 성격).
- Q9 (Out of Scope, 확정): 네이티브 `flutter_web_auth_2` 실 SDK 연동·앱의 state 발급 엔드포인트 호출 배선(운영 셋업), redirect_uri 실제 요구 여부 네이버 공식 문서 최종 확인·실 크레덴셜 등록(운영 작업), GAP-015-01(이미 해소), 카카오·구글 state 메커니즘 신설, state 저장 구체 방식(Planning 위임), 실 네이버 인증 흐름 E2E 검증(015 와 동일 deferred).

- Q16 (보안): state 는 예측 불가능한 값으로 발급, TTL 보관, 콜백 시 1회성 검증·소비. redirect_uri 환경변수 미설정 시 하위 호환 유지.

### [카테고리 7] 예외 & 실패 시나리오

- Q21: 엣지 케이스 — state 재사용(2회째 제출) 거부, 카카오·구글은 state 무관하게 정상 동작.
- Q22 (백업/복구): 해당 없음(state 는 단기 TTL 데이터, 영속 백업 대상 아님).

## 사후 운영 검증 시나리오 (PROC-014, 확정)

1. 실 크레덴셜 등록 + 네이티브 앱이 state 발급 엔드포인트를 호출하도록 배선 완료 후, 실제 state 발급→네이버 인증→콜백→검증 흐름이 정상 동작하는지 확인.
2. state 만료(TTL 경과) 후 재시도 시 사용자에게 적절한 오류 메시지가 표시되고 재시도가 가능한지 확인.
3. redirect_uri 요구 여부를 네이버 공식 문서로 최종 확인 후, 필요 시 `NAVER_REDIRECT_URI` 설정하고 정상 동작 확인.
4. 카카오·구글 로그인 회귀 없음 확인.
