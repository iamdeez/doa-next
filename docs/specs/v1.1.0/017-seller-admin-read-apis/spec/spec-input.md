## 카테고리별 수집 내용

### [카테고리 1] 배경 및 목적
- Q1: `apps/console`(판매자·관리자 콘솔)이 001-skeleton/002-catalog API에 통합하며 발견한 백엔드 계약 갭 6건
  (BE-GAP-002~007, `docs/backend-gaps.md`)을 해소한다. BE-GAP-001은 이미 012에서 해소됨(코드 확인 완료).
- Q2: 현재 관리자는 승인 대기 판매자 목록은 볼 수 있으나(단, 페이지네이션·상태선택·검색 불가) 승인/반려된 판매자
  이력은 조회할 수 없다. 판매자는 DRAFT 상품의 옵션을 되읽을 수 없어 상품 관리 화면이 게시 후에만 정상 동작한다.
  재고 조회·목록 응답 형태가 API마다 비대칭이며, 위시리스트·최근 본 상품은 상품명·가격을 표시할 수 없다.
- Q3 (성공 판단 기준): (1) 관리자가 상태·검색·페이지네이션으로 판매자 목록을 조회할 수 있다, (2) 판매자가 DRAFT
  포함 전체 상태의 자기 상품을 옵션·이미지와 함께 상세 조회할 수 있다, (3) 판매자 상품 목록이 커서 페이지네이션을
  지원한다, (4) 재고 조회·입고 응답이 구조화된 객체다, (5) 위시리스트·최근 본 상품 응답에 상품 요약이 포함된다.

  애플리케이션 내 API 응답 형태·필터·페이지네이션 확장에 한정되므로 Fly.io 배포·컨테이너·네트워크 구성에 영향이
  없다. `infra.md` 확인 결과 관련 기재 불필요 판단(해당 없음).

### [카테고리 7] 예외 & 실패 시나리오
- Q20: 관리자 아닌 사용자의 판매자 목록 조회 → 403(AdminGuard 기존 동작). 비소유 상품 상세 조회 → 403(존재)/
  404(미존재). 존재하지 않는 variantId 재고 조회·입고 → 기존 동작 유지(`getStock`은 미존재 시 0 반환,
  `stockIn`은 `BadRequestException`).
- Q21 (엣지 케이스): 위시리스트/최근 본 상품이 참조하는 상품이 삭제되었거나 DRAFT/INACTIVE 상태 → 항목 유지 +
  "조회 불가" 표시(질문 분석 근거 Q-E 채택). 관리자 판매자 목록 검색어가 빈 문자열 → 필터 미적용(전체 반환).
  페이지네이션 `limit`이 최대값 초과 → 기존 컨벤션(Math.min/Math.max 클램프)과 동일하게 최대값으로 클램프.
- Q22 (백업/복구): 해당 없음 — 신규 영속 데이터 없음(응답 형태·조회 로직 변경만).
