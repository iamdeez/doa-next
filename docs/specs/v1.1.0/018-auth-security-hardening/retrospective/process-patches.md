## PROC-018-01

**PPG-1 세션 한도 중단 시 "부분 산출물 이어받기(증분)" vs "클린 재시작" 재개 프로토콜 명문화**

- **현재 프로세스**: PPG-1(4단계 Development + 5a Test AUTHORING)이 세션 한도로 중단되면 main 이 fresh 재spawn 한다. 그러나 재spawn Agent 가 (A) 중단 시점 부분 산출물을 폐기하고 처음부터 다시 하는지("클린 재시작"), (B) 부분 산출물을 유지하고 git status + tasks.md 체크박스 + 코드 대조로 남은 태스크만 증분 완료하는지("이어받기")의 **분기 기준이 명문화되어 있지 않다**. PROC-015-01(COMPLETE 추정 3조건)은 "중단 Agent 를 COMPLETE 로 볼지" 판정만 다루고, "부분 산출물을 이어받을지 폐기할지"는 다루지 않는다.
- **문제점**: 018 에서 세션 한도 중단이 **2회** 발생했다. 1차 중단은 017 케이스(클린 재시작)와 달리 **양쪽 PPG 에이전트 모두 부분 산출물이 존재**하여 클린 재시작이 불가능했다(중복 구현·산출물 충돌 위험). main 이 "부분 산출물 유지·이어받기"를 즉석 판단했고(pipeline-log L164·L176), 결과적으로 정확히 복구됐다. 나아가 Development 재개 세션조차 T012~T015 를 **run 로그 기록 전에 구현 완료한 채 재차 중단**되어, 재개 Agent 가 "코드는 구현됐으나 run 로그 미기록" 상태를 git+체크박스+코드 3중 대조로 판별해야 했다(run-004 재개분). 이 판단이 매번 즉흥적이면 (a) 부분 산출물을 잘못 폐기해 재작업 낭비, (b) STALE 처리 오판, (c) 이미 구현된 코드를 중복 구현하는 위험이 있다.
- **개선 방향**: orchestration/recovery 문서에 **부분 산출물 이어받기 판정 절차**를 추가한다:
  1. 재spawn 전 main 이 중단 시점 git status(working tree 변경 파일) + 해당 단계 tasks.md 체크박스 상태를 확인한다.
  2. **부분 산출물 존재(working tree 에 해당 단계 변경 파일 있음) → "이어받기" 모드**로 재spawn(MOD=재개). 재spawn prompt 에 "부분 산출물 유지·증분 완료" 지시 + "run 로그 미기록 코드 존재 가능 — git+tasks 체크박스+코드 3중 대조로 기구현 태스크 식별" 지시를 명시한다.
  3. **부분 산출물 없음(working tree 클린) → "클린 재시작" 모드**(017 케이스).
  4. 이어받기 모드에서는 STALE 처리를 적용하지 않는다(중단 코드는 폐기 대상이 아님). 재spawn Agent 는 §6.2 시작 절차 단축(MOD=재개)을 적용하되, 체크 6(복귀 확인)에서 3중 대조를 필수 수행한다.
- **영향 범위**: `~/.claude/docs/pipeline-recovery.md`(재개 절차 — 부분 산출물 이어받기 판정 표 추가), `~/.claude/skills/pipeline/SKILL.md`(main 재spawn 전 git status+체크박스 확인 단계), 부수적으로 `~/.claude/agents/04-development.md`·`05-test.md`(이어받기 모드 시 3중 대조 명시 — 이미 실무에서 준수했으나 명문화).
- **적합성**: 범용 O(모든 PPG 병렬 단계의 세션 한도 중단 복구에 적용) / 역할정합 O(프로세스·흐름 제어·복구 개선)
- **관찰 근거**: 018 세션 한도 2회 중단 — 부분 산출물 이어받기가 실무적으로 성공(모범 사례)했으나 절차가 즉흥적. PROC-015-01(COMPLETE 추정)과 인접·보완 관계. 018 1회 관찰이나 세션 한도 중단은 본 환경에서 반복 관찰되는 특성(017·018 연속)이며 복구 정확성이 데이터 무결성에 직결되므로 안전망 도출을 우선.
- **상태**: 적용 완료 (2026-07-04, main session — `~/.claude/docs/pipeline-recovery.md` §1.3.2 신규, 변경 로그: `~/.claude/docs-change-logs/2026-07-04-002.md`)
- **PROC-018-02(context.md §7 갱신이력 구조 결정)**: 보류 — 단일 문서 구조 변경 결정으로 별도 논의 위임(사용자 미채택).

## PROC-018-02

**context.md §7 "갱신 이력" 섹션의 반쪽 유지(orphan) 상태 해소 — 섹션 제거 vs 일관 유지 결정**

- **현재 프로세스**: 이 프로젝트의 `context.md` 는 전역 템플릿(`00-context-rules.md §6`)을 따라 "§7. 갱신 이력" changelog 성 표를 보유한다. 그러나 PROC-R02(retrospective agent 정의)는 changelog 성 섹션에 신규 spec 행 추가를 금지하며, 변경 추적 SoT 를 git history + `docs-change-logs/` + `docs/specs/` 폴더 구조로 규정한다.
- **문제점**: 두 규정이 충돌한다. 그 결과 §7 갱신 이력은 **016 까지만 기록되고 017·018 은 미기록**인 반쪽 유지(orphan) 상태다 — 017 retrospective 가 PROC-R02 를 적용해 §7 행을 추가하지 않았고, 018 도 동일하게 추가하지 않는다(agent-patches.md §7 처리 결정 참조). §7 이 "016 이 마지막 spec 인 것처럼" 보이는 오해를 유발하며(실제 018 까지 진행), changelog 부재-공지성 잔재로 남는다.
- **개선 방향**: 다음 중 하나로 §7 처리를 확정한다(사용자 결정):
  - **(A, 권고) §7 갱신 이력 섹션 제거**: PROC-R02·전역 CLAUDE.md(changelog 부재는 git history + docs-change-logs SoT) 원칙에 정합. 제거 시 markdown.md "부재-공지성 메타 주석 금지"에 따라 "이력 없음" 류 대체 주석을 남기지 않는다. 이 경우 전역 템플릿(`00-context-rules.md §6`)에서도 §7 을 제거하거나 "선택" 표기하는 후속 전역 패치가 필요하다(템플릿-실무 정합).
  - **(B) §7 일관 유지**: 매 spec 완료 시 §7 행을 추가하도록 복원. 단 PROC-R02 와 정면 충돌하므로 PROC-R02 완화가 선행되어야 한다.
- **영향 범위**: `{project}/.claude/docs/context.md`(§7 처리), `~/.claude/rules/specs/00-context-rules.md §6`(템플릿의 §7 정의 — A 채택 시 정합화), `~/.claude/agents/07-retrospective.md`(PROC-R02 문구 — A/B 결정에 따라 조정).
- **적합성**: 범용 O(context.md 를 보유한 모든 프로젝트의 §7 처리 원칙에 영향 — 전역 템플릿 정합) / 역할정합 O(문서 구조·작성 원칙 개선)
- **관찰 근거**: 017·018 **2회 연속** §7 행 미추가로 orphan 상태 심화. GAP-018-03(Docs)이 §7 018 행 추가를 요청한 것 자체가 PROC-R02 와 실무 인식의 격차를 드러낸다 — 규정 충돌의 항구적 해소 필요.
- **상태**: 후보 (사용자 승인 대기 — A 권고)
