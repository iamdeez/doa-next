## 배경 및 목적

001~020 차수에 걸쳐 18개 도메인 모듈이 실구현되었으나, 결제와 파일 저장소 두 외부 연동
지점은 여전히 **stub 구현**으로 남아 있다.

- `PaymentGatewayPort`(`apps/backend/src/modules/payment/payment-gateway.port.ts`)는
  `charge`/`refund` 2개 메서드를 정의하고 있으나, 현재 바인딩된 `StubPaymentGateway`
  (`payment.module.ts`)는 항상 성공을 반환하는 가짜 트랜잭션 ID(`stub-tx-{key}`)만 생성한다.
  실제 PG(Payment Gateway) 승인·거절·환불이 발생하지 않는다.
- `FileStoragePort`(`apps/backend/src/modules/file/file-storage.port.ts`)는
  `getPresignedUploadUrl`/`getPublicUrl` 2개 메서드를 정의하고 있으나, 현재 바인딩된
  `StubFileStorage`(`file.module.ts`)는 무네트워크 결정적 URL만 반환한다. 실제 Cloudflare R2
  객체 저장소에 파일이 저장·서빙되지 않는다.

두 Port 모두 constitution P-002(AWS 의존 금지)·P-004(클라우드 중립) 원칙 하에 실 구현체 교체만
으로 전환 가능하도록 설계되어 있다(020-data-migration-cutover CHANGES.md "미완 선행 조건" 항목
3 에서 두 연동을 독립된 후속 spec 으로 명시). 본 spec 은 이 두 지점을 각각 **KG이니시스**(결제)와
**Cloudflare R2**(파일)로 실연동하여, 주문·정산·리뷰 등 이미 실구현된 도메인 흐름이 실제 금전
거래·실제 파일 저장을 통해 동작하도록 한다.

본 spec 완료의 목표는 "코드 연동 완료 + 이니시스 sandbox(공용 테스트 상점ID)·R2 실 버킷을
통한 실제 API 호출 검증"까지이며, 실 가맹점 계약(MID) 발급 후 운영(prod) 전환은 범위 외 후속
과제다(사업자 계약이 현재 진행 중 — ASM-003).

> **결제수단 범위 정정 이력**: 사용자는 최초 결제수단 범위를 "전체(신용카드·계좌이체·가상계좌·
> 간편결제)"로 확정했으나, 가상계좌 결제는 구조적으로 비동기 입금통지(웹훅) 수신 인프라를
> 요구한다는 설명을 들은 뒤 이를 정정 — **가상계좌를 범위에서 제외**하고 웹훅 요구사항도
> 철회했다. 본 spec 의 결제수단 범위는 **신용카드·실시간 계좌이체·간편결제**(이니시스 지원
> 범위 내) 3종이며, 모든 결제는 요청에 대한 **동기 승인 응답**으로 완결된다(비동기 완료 흐름
> 없음). 가상계좌·웹훅은 "범위 외" 절 참조.
>
> **PG 장애 시 재시도 방식 정정 이력(GAP-021-01)**: 최초 spec 은 "PG API 호출 실패(타임아웃·
> 5xx) 시 기존 `payment_outbox` 재시도 흐름으로 처리"(FR-004/SC-007/ASM-009)로 서술했으나,
> Planning Agent 가 실제 `payment_outbox` 아키텍처를 확인한 결과 이 구조는 **성공한 결제완료
> 이벤트(payment.completed 등)를 pg-boss 로 relay 하는 용도**이며 charge 실패 시 레코드를
> 생성하거나 재시도하는 메커니즘 자체가 없어 최초 서술과 모순됨을 발견(GAP-021-01,
> CLARIFICATION_NEEDED). 사용자가 **옵션 1(멱등키 재요청 방식)**을 승인 — 신규 재시도 코드를
> 추가하지 않고, PG 실패 시 결제 상태를 `failed` 로 기록한 뒤 **클라이언트가 동일 멱등성
> 키(idempotency key)로 재요청하면 중복 결제 없이 안전하게 처리**되는 기존 멱등성 계약(P-005)
> 만으로 재시도를 해결한다. FR-004·SC-007·ASM-009 를 이 방식으로 정정했다.

## 사용자 스토리

- **US-001**: 구매자로서, 신용카드·실시간 계좌이체·간편결제로 실제 결제를 수행하여 주문을
  확정하고 싶다.
- **US-002**: 구매자로서, 주문 취소 시 실제 PG를 통한 환불(전액 또는 부분)이 처리되길 원한다.
- **US-003**: 판매자·구매자로서, 상품 이미지·리뷰 이미지 등을 업로드하면 실제 클라우드
  저장소(R2)에 저장되고 공개 URL로 조회 가능하길 원한다.
- **US-004**: 플랫폼 운영자로서, PG 장애·타임아웃이 발생해도 결제 상태가 안전하게 `failed` 로
  기록되고, 클라이언트가 동일 멱등키로 재요청하면 중복 결제 없이 정합성이 유지되길 원한다.
- **US-005**: 개발자/QA로서, 실 가맹점 계약 없이도 이니시스 sandbox 자격증명으로 실제
  결제·환불 API 호출 흐름을 검증할 수 있길 원한다.

## 기능 요구사항

### Track A — 결제 게이트웨이(KG이니시스) 실연동

- **FR-001**: `PaymentGatewayPort` 구현체는 KG이니시스 API를 통해 신용카드·실시간 계좌이체·
  간편결제(이니시스 지원 범위 내) 결제 요청을 실제로 전달하고, **동기 응답**으로 승인 또는
  거절 결과를 수신해야 한다(가상계좌·비동기 완료 흐름은 범위 외).
- **FR-002**: `PaymentGatewayPort` 구현체는 KG이니시스 API를 통해 실제 환불(전액·부분) 요청을
  전달해야 한다.
- **FR-003**: 결제·환불 요청은 멱등성 키를 KG이니시스 API 호출 페이로드에 포함해야 한다.
- **FR-004**: KG이니시스 API 호출이 실패(타임아웃·5xx 등)하는 경우, 시스템은 결제 상태를
  `failed` 로 기록해야 하며, 클라이언트가 동일 멱등성 키(idempotency key)로 재요청하면
  중복 결제 없이 안전하게 처리(재시도)되어야 한다(신규 재시도 메커니즘·`payment_outbox`
  활용 없이 기존 멱등성 계약만으로 해결 — GAP-021-01 정정).
- **FR-005**: 시스템은 이니시스 sandbox(공용 테스트 상점ID) 자격증명과 향후 운영(실 MID)
  자격증명을 코드 변경 없이 설정(환경변수 등) 전환만으로 교체할 수 있어야 한다.
- **FR-006**: 환불 승인은 기본적으로 자동 처리되어야 하며, 관리자 개입이 필요한 예외 케이스가
  발생하면 기존 관리자 감사로그(`admin_audit_logs`) 패턴을 재사용해야 한다.

### Track B — 파일 저장소(Cloudflare R2) 실연동

- **FR-007**: `FileStoragePort` 구현체는 Cloudflare R2에 대해 실제 presigned upload URL을
  발급해야 한다.
- **FR-008**: `FileStoragePort` 구현체는 R2에 저장된 객체의 실제 공개 URL(r2.dev 서브도메인
  기반)을 반환해야 한다.
- **FR-009**: 실 연동 전환 후에도 기존 이미지 4종 allowlist(011-file-security)·10MiB 업로드
  상한 검증 로직이 회귀 없이 유지되어야 한다.

## 비기능 요구사항

- **NFR-001** (성능): 결제 관련 API(`POST /payments` 등) 응답 P95 는 2,000ms 이하를 유지해야
  한다(기존 infra.md NFR-002 승계). 모든 결제가 동기 승인 흐름이므로 KG이니시스 외부 API 호출
  지연이 API 응답시간에 직접 포함되며, 이는 시스템이 직접 통제할 수 없는 변동 요인으로
  인지한다(ASM-004).
- **NFR-002** (정합성): 결제·환불 **완료(성공)** 상태 변경 및 후속 도메인 이벤트 발행은
  outbox 패턴 없이 처리되어서는 안 되며, 모든 결제·환불 요청은 멱등성 키 없이 처리되어서는
  안 된다(constitution P-005 승계). PG 실패(`failed`) 기록 자체는 outbox 대상이 아니며
  멱등키 기반 재요청으로 안전성을 보장한다(FR-004, GAP-021-01 정정 반영).
- **NFR-003** (데이터 무결성): 결제·환불 관련 신규 금전 연산은 Prisma `Decimal` 타입만
  사용해야 하며 부동소수점 연산을 사용하지 않아야 한다(constitution P-005 승계).
- **NFR-004** (로깅 보안): 결제 요청/응답 로그 및 이니시스 자격증명(MID·API 키 등)·카드 관련
  정보는 평문으로 로그에 노출되지 않아야 한다(018/019 pino redact 패턴 재사용).

## 범위 외

- **가상계좌 결제 및 비동기 입금통지(웹훅) 처리**: 사용자가 최초 "결제수단 전체(가상계좌
  포함)"를 요청했으나, 가상계좌는 구조적으로 비동기 웹훅 수신 인프라를 요구한다는 설명을 들은
  뒤 이를 정정하여 **제외 확정**했다. 본 spec 은 동기 승인 결제(카드·실시간 계좌이체·
  간편결제)만 다루며, 웹훅 엔드포인트·비동기 결제완료 처리 로직은 구현하지 않는다. 향후
  가상계좌 지원이 필요하면 별도 spec 으로 진행한다.
- **실 가맹점(운영 MID) 전환**: 사업자 계약이 진행 중이라 실 MID 를 사용한 운영 결제 전환은
  본 spec 범위 밖이다(ASM-003). FR-005 가 요구하는 "설정 전환만으로 교체 가능"한 구조까지만
  본 spec 이 다룬다.
- **PG사 이중화·다중 벤더**: KG이니시스 외 타 PG사(토스페이먼츠 등) 동시 지원은 범위 밖이다.
- **정기결제/구독 결제**: 사용자 요청에 명시되지 않은 인접 기능으로, 본 spec 은 1회성
  주문 결제·환불만 다룬다.
- **R2 커스텀 도메인**: 공개 URL 은 r2.dev 서브도메인만 사용하며, Cloudflare 커스텀 도메인
  설정은 범위 밖이다(사용자 확정).
- **파일 바이너리 이관(020 대기 항목)**: 020-data-migration-cutover 가 R2 실연동 완료를
  전제로 별도 이월시킨 "레거시 파일 바이너리 이관"은 본 spec 에 포함하지 않는다. 본 spec
  완료 후 별도 spec 생성 여부는 main session 이 020 완료 시점에 판단한다.
- **`POST /files/:id/confirm` 의 실 R2 HEAD 교차검증**: context.md §6 에 011-file-security 의
  잔여 항목으로 "confirm size 는 클라이언트 신뢰값 — 실 R2 HEAD 교차검증은 후속"이 기록되어
  있다. 본 spec 은 R2 를 실제로 연동하지만, 사용자가 이 교차검증 강화를 명시적으로 요청하지
  않았으므로 범위 외로 유지한다(향후 별도 spec 후보로 남김).
- **PCI-DSS 정식 컴플라이언스 감사**: 이니시스 표준(호스팅) 결제창 사용을 가정하여 카드번호
  비수집을 전제(ASM-006)로 하나, 정식 PCI-DSS 인증·감사 절차 자체는 범위 밖이다.
- **용량 설계(트래픽 규모 기반)**: 초기 트래픽 규모가 미정(ASM-007)이므로 특정 동시 결제
  건수·업로드량을 전제로 한 용량 설계는 범위 밖이다.

### 사후 운영 검증 피드백 사이클 (PROC-014)

본 spec 파이프라인 종료 후 사용자가 운영 환경(sandbox 또는 실 MID 전환 이후)에서 점검할
가능성이 있는 시나리오:

1. 실 MID 발급 후 첫 실결제(신용카드·실시간 계좌이체·간편결제 각 1건)가 sandbox 검증과
   동일하게 동작하는지.
2. R2 실 버킷에서 대량 동시 업로드 시 presigned URL 발급 지연·만료 이슈가 없는지.
3. (향후 가상계좌 지원 재검토 시) 비동기 입금통지 인프라 도입 필요성 재평가.

결함 발견 시 처리 절차: 결함 정보를 본 spec.md "배경 및 목적" 절 입력 또는 별도 patch spec
입력으로 사용 → main session 의 "spec 수정" 이벤트 → 1단계 재진입 또는 별도 patch spec 진입.
사후 검증 미수행 시에는 실 MID 발급 시점을 차후 점검 일자로 삼는다(사용자와 별도 합의 없음 —
사업자 계약 완료 시점이 곧 트리거).

## 미결 사항

없음 — 사용자 확인을 거쳐 모든 핵심 불확실성(PG 벤더·결제수단 범위·범위 경계·크레덴셜
상태·가상계좌 제외 정정)이 해소되었다. 잔존 항목(가맹점 명의 [TO-VERIFY] 등)은 ASM-XXX 로
`assumptions.md` 에 문서화되어 있으며 본 spec 진행을 차단하지 않는다.
