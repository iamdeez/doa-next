## 카테고리별 수집 내용

### [카테고리 1] 배경 및 목적
- Q1(성공기준이 실운영 개시 전제인지): 아니오 — sandbox 검증까지가 목표. 실 MID 발급 후 실 운영 전환은 범위 외(후속).
- Q2(현재 한계 확인): 확인됨 — StubPaymentGateway(항상 성공)·StubFileStorage(무네트워크 결정적 URL)로 실제 결제·업로드 미발생.
- Q3(성공 판단 기준): "PG sandbox 를 통한 실제 charge/refund 승인 성공 + 리턴 데이터 기반 상태 처리 검증 + R2 실 버킷 업로드·공개 URL 접근 성공"(ASM-001).

### [카테고리 2] 사용자 & 이해관계자
- Q4(추가 이해관계자): 없음 — 기존과 동일(구매자·판매자·플랫폼 운영자)(ASM-002).
- Q5(기술 수준): 기존과 동일 가정, 별도 확인 불필요.
- Q6(가맹점 명의): 미상 — [TO-VERIFY](ASM-003, 실 MID 발급 시점에 확인. sandbox 단계는 무관하여 비차단).

### [카테고리 3] 핵심 기능
- Q7(PG 벤더/결제수단 범위): **KG이니시스** 직접 연동. 결제수단 최초 **전체**(신용카드·계좌이체·가상계좌·간편결제 — 이니시스 지원 범위 내) 확정 → **[정정] 가상계좌 제외**, 최종 범위는 **신용카드·실시간 계좌이체·간편결제** 3종(사용자가 가상계좌의 웹훅 필요성 설명을 들은 뒤 정정). MID 미확보 → 이니시스 공용 테스트 상점ID 로 sandbox 개발.
- Q8(R2 파일 정책 유지 여부): 기존 이미지 4종 allowlist·10MiB 상한 **유지**.
- Q9(범위 경계): **(B)** — PaymentGatewayPort/FileStoragePort 구현체 교체(코드) + 실제 sandbox 크레덴셜로 charge/refund 호출·PG 리턴 데이터 기반 상태 처리까지 검증. 실 운영(prod MID) 전환은 범위 외. 020 파일 바이너리 이관은 본 spec 미포함 — 021 완료 후 별도 spec 여부는 main 이 020 완료 시점에 판단.

### [카테고리 4] 데이터 & 입출력
- Q10(PG 크레덴셜 상태): 사업자 계약 **진행 중**(미완료) — MID 미공유. sandbox 공용 테스트 상점ID로 대체.
- Q11(웹훅/콜백): 최초 **포함 필수로 확정**(가상계좌 비동기 입금통지 목적) → **[정정] 철회**. 가상계좌 제외 확정에 따라 웹훅 수신 엔드포인트·비동기 완료 처리 로직 불요. 모든 결제는 동기 승인 흐름으로 완결(FR-001).
- Q12(R2 계정·버킷 상태): 아직 미생성. 사용자가 Development 단계 직전 직접 생성(Cloudflare 가입→R2 활성화→버킷 생성→r2.dev public access→API 토큰 발급) — spec.md 에 전제로 명시(ASM-013).

### [카테고리 3·4 정정 이력]

SC-004~008 등) 포함 구조로 작성했으나, 사용자가 이어서 "가상계좌 제외·웹훅 철회"로 정정

재구성했다. 정정 사유: 가상계좌만이 비동기 웹훅을 요구하는 유일한 결제수단이었으므로, 가상계좌
제외 시 웹훅 요구사항 자체가 소멸.

### [카테고리 5] 제약조건

- Q14(성능): NFR(결제 API P95 ≤2,000ms) 유지, PG 응답시간 변동성을 리스크로 명시(ASM-004).

### [카테고리 7] 예외 & 실패 시나리오
- Q20(PG 장애 시): 최초 "기존 payment_outbox 재시도 흐름 재사용"(ASM-009 1차, 적합 판정) →
  **[정정, GAP-021-01]** Planning Agent 가 실제 `payment_outbox` 아키텍처(성공한 결제완료
  이벤트 relay 전용, charge 실패 레코드·재시도 메커니즘 없음)를 확인하여 CLARIFICATION_NEEDED
  로 보고. 사용자가 **옵션 1(멱등키 재요청 방식)**을 승인 — PG 실패 시 결제 상태를 `failed`
  로 기록하고 클라이언트가 동일 멱등키로 재요청하면 안전하게 재시도되는 방식으로 확정
  (신규 서버측 재시도 코드 0, 기존 outbox 구조 무변경). ASM-009·FR-004·SC-007 정정 반영.
- Q21(R2 업로드 실패): 클라이언트 재시도 책임 유지 — presigned URL 모델 특성(ASM-010).
- Q22(백업/복구): 기존 Fly Postgres 백업 정책 재사용, 추가 요구 없음(ASM-011).

### [카테고리 7 정정 이력(GAP-021-01)]
Planning Agent 가 SC-007/FR-004/ASM-009(PG 장애 시 `payment_outbox` 재시도) 가 실제
`payment_outbox` 아키텍처와 모순됨을 CLARIFICATION_NEEDED 로 보고했다 — 해당 outbox 는
성공한 결제완료 이벤트를 pg-boss 로 relay 하는 용도이며, charge 실패 시 레코드를 생성하거나
재시도하는 메커니즘이 없다. 사용자가 옵션 1(멱등키 재요청 방식 — 신규 코드 0, 기존 outbox
구조 무변경)을 승인하여 spec.md 를 v1.2 로 갱신(FR-004·SC-007·NFR-002·SC-014 문구 정정,
assumptions.md ASM-009 정정). Track A/B 의 다른 FR·NFR·SC·범위외 항목은 변경 없음.
