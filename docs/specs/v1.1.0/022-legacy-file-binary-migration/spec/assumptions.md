# Assumptions: 022-legacy-file-binary-migration

| ID | 가정 내용 | 확인 필요 여부 | 확인 방법 |
|---|---|---|---|
| ASM-001 | 레거시 파일 바이너리는 AWS S3(또는 S3 호환 오브젝트 스토리지)에 저장되어 있다고 가정한다. 실제 스토리지 서비스·버킷 구조는 파이프라인 밖(사용자 환경)이라 [TO-VERIFY]다. | 필요 (실 이관 실행 전) | 사용자가 레거시 환경에서 버킷명·리전·엔드포인트를 확인(옵션 A) |
| ASM-002 | 레거시 객체 key 네이밍이 R2 객체 key 로 그대로 사용 가능한 형태(경로 문자·길이 제약 등 S3 호환 API 제약 내)라고 가정한다(FR-004). | 필요 | 사전 대량 복사 실행 시 key 유효성 오류 발생 여부로 확인 — 오류 시 FR-004 재작업 |
| ASM-003 | 체크섬/해시 비교(FR-005·SC-005)는 오브젝트 스토리지가 표준으로 제공하는 ETag(또는 MD5) 기반으로 가능하다고 가정한다. 레거시가 멀티파트 업로드를 사용한 객체는 ETag 가 단순 MD5 가 아닐 수 있어(S3 멀티파트 ETag 는 파트 해시들의 해시), 이 경우 별도 콘텐츠 해시 재계산이 필요할 수 있다. | 필요 (검증 단계 설계 시) | 사전 실측(FR-009) 단계에서 샘플 객체의 ETag 형식 확인 |
| ASM-004 | 레거시 파일 총 개수·총 용량에 대한 사전 추정치가 없어 FR-009 사전 실측(precheck) 단계로 대체한다. 실측 결과가 NFR-001(60분 델타 윈도우)에 위협적인 수준(대량 델타 발생)이 아닐 것으로 가정한다. | 필요 (사전 실측 완료 시점) | FR-009 사전 평가 리포트 실측값 확인 — 위협 수준이면 FR-007 경로로 사용자 재확인 |
| ASM-005 | `files.files.status=UPLOADED` 레코드는 레거시 시스템에도 대응하는 실제 바이너리 객체가 존재한다고 가정한다(020 이 이관한 메타데이터의 UPLOADED 상태가 실제 업로드 완료를 신뢰성 있게 반영). | 필요 | SC-001 개수 대조 시 레거시 실측 개수와 `files.files`(status=UPLOADED) 레코드 수 불일치 여부로 확인 — 불일치 시 원인 조사(고아 메타데이터 또는 고아 객체) |

> 모든 항목은 본 spec 진행을 차단하지 않으며(FR-008 옵션 A 원칙), 실 이관 실행 시점에 사용자 환경에서 실측·확인된다.
