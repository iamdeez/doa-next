## 배경 및 목적

020-data-migration-cutover 는 레거시 AWS RDS(18개 서비스) → 신규 Fly Postgres(8스키마 33테이블) 데이터 이관·컷오버 절차를 설계했으나, `files.files` 테이블에 대해서는 **메타데이터**(id·ownerId·purpose·key·url·contentType·size·status·createdAt)만 이관 대상에 포함했다(020 FR-017/SC-019). 실 파일 바이너리(오브젝트) 자체의 이관은 "R2 실연동 완료 후 별도 진행"을 전제로 명시적으로 범위 외 처리되었다(020 spec.md "범위 외" 절, 020 CHANGES.md 후속 주의사항 #3).

021-payment-file-integration 이 `FileStoragePort` 의 실 구현체 `R2FileStorage`(`apps/backend/src/modules/file/r2-file-storage.ts` — `@aws-sdk/client-s3` 기반 Cloudflare R2 presigned URL 발급)를 완성하여 이 선행 조건이 충족되었다(021 CHANGES.md 후속 주의사항 #6).

그 결과 현재 남은 공백은: 레거시 AWS(S3 등)에 저장되어 있는 실제 파일 바이너리(상품 이미지·리뷰 이미지·프로필 사진 등)가 아직 신규 Cloudflare R2 버킷에 존재하지 않는다는 점이다. 020 이 이관한 `files.files` 메타데이터의 `url` 필드는 실제 객체가 없는 상태로 컷오버될 경우, 신규 시스템에서 파일 접근이 깨진다(404). 본 spec 은 이 실 바이너리 이관을 **무엇을, 왜** 수행해야 하는지 정의한다. 구체적인 스크립트 구현·병렬화 전략 등은 plan.md(HOW)로 위임한다.

## 사용자 스토리

- **US-001**: 운영자로서, 레거시 AWS S3 에 있는 실 파일 바이너리를 020 컷오버 유지보수 윈도우 이전에 시간 제약 없이 사전 대량 복사하여, 컷오버 당일 처리량을 최소화하고 싶다.
- **US-002**: 운영자로서, 컷오버 유지보수 윈도우 동안에는 사전 복사 완료 시점 이후 발생한 최종 델타 파일만 추가로 이관하여, 60분 윈도우 내에서 파일 데이터 정합성을 확보하고 싶다.
- **US-003**: 운영자로서, 이관된 파일 바이너리가 레거시와 정확히 일치함을 자동화된 검증(개수 대조 + 샘플 체크섬)으로 확인하여 데이터 유실 없이 전환되었음을 확신하고 싶다.
- **US-004**: 운영자로서, 개별 파일 이관 실패가 전체 컷오버 절차를 막지 않고 실패 목록으로 추적·재시도되길 원한다.
- **US-005**: 구매자·판매자로서, 컷오버 이후에도 기존에 업로드한 상품·리뷰·프로필 이미지에 변함없이(URL 안정적으로) 접근하고 싶다.

## 기능 요구사항

- **FR-001**: 이관 절차는 `files.files` 테이블에서 `status=UPLOADED` 인 레코드가 참조하는 실 파일 바이너리(오브젝트)를 레거시 AWS S3(또는 등가 오브젝트 스토리지)에서 Cloudflare R2 로 복사해야 한다. `status=PENDING` 레코드(업로드 미완료로 추정)는 이관 대상에서 제외한다.
- **FR-002**: 이관 절차는 020 컷오버의 유지보수 윈도우 개시 전에 대상 파일의 벌크 사전 복사를 완료해야 한다.
- **FR-003**: 유지보수 윈도우 개시 이후에는 사전 복사 완료 시점 이후 신규로 `UPLOADED` 상태가 된 파일(최종 델타)만 추가로 이관해야 한다.
- **FR-004**: 이관된 각 파일 객체는 레거시 객체 key 값을 그대로 R2 객체 key 로 사용해야 하며, 해당 `files.files` 레코드의 `url` 필드만 R2 공개 URL(`R2_PUBLIC_BASE_URL` + key) 로 갱신해야 한다(키 값 자체의 재작성 없음).
- **FR-005**: 이관 완료 후 레거시-신규(R2) 간 다음 검증을 수행해야 한다: (a) 이관 대상 파일 개수 100% 일치, (b) 무작위 샘플에 대한 체크섬/해시(예: ETag/MD5) 비교 일치.
- **FR-006**: 개별 파일 객체 이관이 실패하는 경우, 절차는 전체 이관을 중단하지 않고 실패 목록을 기록하여 재시도해야 한다(부분 성공 허용).
- **FR-007**: 사전 복사 단계에서 재시도 후에도 해소되지 않는 실패가 컷오버 유지보수 윈도우 개시 시점까지 잔존하면, 그 사실을 사전 평가 리포트(FR-009)에 포함하여 사용자 재확인을 요청해야 한다.
- **FR-008**: 레거시 AWS S3 접근(자격증명·버킷·네트워크)은 파이프라인 밖 사용자 환경에서만 가능하므로, 이관 절차는 실행 스크립트·명령을 산출물로 제공하고 사용자가 실 환경에서 실행한 결과를 전달받아 검증하는 방식(옵션 A)으로 진행해야 한다.
- **FR-009**: 이관 절차는 실행 전 레거시 파일 총 개수·총 용량을 측정하는 사전 평가(precheck) 단계를 포함해야 한다.
- **FR-010**: 이관 실행 인프라는 020 이 구축한 전용 러너(Fly one-off machine, `scripts/migration/Dockerfile`)를 재사용 또는 확장해야 한다(신규 별도 이미지 도입 없음).

## 비기능 요구사항

- **NFR-001** (시간 — 델타 구간): 컷오버 유지보수 윈도우 내에서 처리되는 최종 델타 파일 이관은 020 NFR-001(윈도우 60분) 제약 내에서 완료되어야 한다.
- **NFR-002** (시간 — 사전 복사 구간): 사전 대량 복사 단계는 별도 시간 상한을 두지 않는다.
- **NFR-003** (정합성): 이관 완료 시점의 파일 개수 대조는 레거시-신규 간 불일치 0건이어야 한다.
- **NFR-004** (전송 보안): 파일 전송 채널은 TLS/HTTPS 를 사용해야 한다.
- **NFR-005** (감사 가능성): 이관 절차 실행 이력(개별 파일 성공/실패, 재시도 횟수, 검증 결과)은 감사 가능한 형태로 기록되어야 한다.

## 수용 기준

- **SC-001** (`FR-001` 관련): 이관 파이프라인 실행 후 `status=UPLOADED` 레코드가 참조하는 파일 객체 전건이 R2 버킷에 존재하고, `status=PENDING` 레코드는 이관 대상에서 제외되었음을 확인한다. [env:e2e-db]
- **SC-002** (`FR-002` 관련): 벌크 사전 복사 완료 시점에 사전 복사 대상 파일의 100%가 R2 로 이관(성공)되었음을 확인한다(잔존 실패는 SC-007 로 별도 처리). [env:e2e-db]
- **SC-003** (`FR-003` 관련): 컷오버 윈도우 개시 이후, 사전 복사 완료 시점 이후 생성된 최종 델타 파일(신규 `UPLOADED` 레코드)이 윈도우 내에 추가 이관됨을 확인한다. [env:e2e-db]
- **SC-004** (`FR-004` 관련): 이관된 각 R2 객체의 key 값이 레거시 객체 key 값과 동일하며, 해당 `files.files` 레코드의 `url` 필드가 `R2_PUBLIC_BASE_URL` + key 형태로 갱신되어 있음을 확인한다. [env:integration]
- **SC-005** (`FR-005`·`NFR-003` 관련): 정합성 검증 리포트에서 이관 대상 파일 개수가 레거시=신규로 100% 일치하고, 무작위 샘플(스키마당 최소 100건 또는 전체 1% 중 큰 값, 020 SC-007 패턴 승계)의 체크섬이 일치함을 확인한다. [env:e2e-db]
- **SC-006** (`FR-006` 관련): 개별 파일 이관 실패를 인위적으로 재현했을 때, 절차가 전체를 중단하지 않고 실패 목록에 기록한 뒤 재시도를 수행함을 확인한다. [env:integration]
- **SC-007** (`FR-007` 관련): 사전 복사 완료 후에도 미해결 실패가 잔존하는 시나리오를 재현했을 때, 사전 평가 리포트에 잔존 실패 목록과 "컷오버 개시 전 사용자 재확인 필요" 문구가 포함됨을 확인한다. [env:static]
- **SC-008** (`FR-008` 관련): 실행 절차 문서(런북)에 레거시 S3 자격증명·버킷 접근이 필요한 단계마다 "사용자 환경 실행 → 결과 전달 → 검증" 절차가 명시되어 있음을 확인한다. [env:static]
- **SC-009** (`FR-009` 관련): 사전 평가 리포트에 레거시 파일 총 개수·총 용량 실측값과 예상 소요 시간이 기재되어 있음을 확인한다. [env:static]
- **SC-010** (`FR-010` 관련): 파일 이관 스크립트가 020 전용 러너 이미지(`scripts/migration/Dockerfile`) 내에서 정상 실행 가능함을 확인한다(신규 별도 이미지 생성 없음). [env:static]
- **SC-011** (`NFR-001` 관련): 리허설 또는 실제 실행 로그에서 컷오버 윈도우 내 델타 파일 이관 소요가 020 NFR-001(60분) 범위 안에 포함됨을 확인한다. [env:e2e-db]
- **SC-012** (`NFR-002` 관련): 사전 대량 복사 단계 실행 스크립트·로그에 별도 시간 상한 강제 로직(타임아웃에 의한 자동 중단)이 없음을 확인한다. [env:static]
- **SC-013** (`NFR-003` 관련): 컷오버 개시 시점(사전 복사 완료 확정 시점) 파일 개수 대조에서 불일치 0건임을 확인한다. [env:e2e-db]
- **SC-014** (`NFR-004` 관련): 이관 전송 채널 설정에 TLS/HTTPS 가 적용되어 있음을 설정 검토로 확인한다. [env:static]
- **SC-015** (`NFR-005` 관련): 이관 실행 로그(파일별 성공/실패·재시도 횟수·검증 결과)가 감사 가능한 형태(파일 또는 감사 테이블)로 저장됨을 확인한다. [env:integration]

## 범위 외

- **레거시 AWS S3 실제 해체(삭제)**: 컷오버 이후 레거시 오브젝트 스토리지를 실제로 삭제·해체하는 절차는 본 spec 범위 외다(020 "레거시 AWS 인프라 해체" 범위 외 원칙 승계) — 별도 후속 spec.
- **Cloudflare "Super Slurper" 등 웹 콘솔 기반 벌크 이관 서비스 채택**: 사용자가 서버사이드 스크립트 방식(옵션 A)을 확정 선택함에 따라, 콘솔 조작 기반 이관 도구는 범위 외다.
- **`files.files.key` 값 리키잉(재포맷)**: 이관된 레코드의 key 값을 신규 포맷(`{purpose}/{ownerId}/{uuid}`)으로 재작성하는 것은 범위 외다(사용자가 최소 변경 원칙에 따라 기존 key 유지를 확정). 향후 필요 시 별도 spec.
- **`status=PENDING` 레코드의 바이너리 이관**: 업로드 미완료로 추정되는 레코드는 대상에서 제외한다(FR-001).
- **`POST /files/:id/confirm` 의 실 R2 HEAD 교차검증 강화**: `context.md §6`·021 spec "범위 외" 절이 이미 이월한 항목으로, 본 spec 도 동일하게 범위 외로 유지한다.
- **point of no return 이후 자동 롤백**: 020 FR-008 이 정의한 PoNR(신규 주문/결제 발생 시점) 원칙을 그대로 승계하며, 파일 바이너리에 대한 별도의 자동 역이관 기능도 지원하지 않는다.
- **Fly Postgres/인프라 HA 구성 여부 결정**: 020 이 범위 외로 이월한 인프라 결정(`infra.md §8`)에 대해 본 spec 은 관여하지 않는다.
- **사후 검증 활동 실행 방식**: 실제 레거시 AWS S3 접속이 필요한 검증(SC-001·002·005·007·009·011·013)은 파이프라인 내에서 자동 실행할 수 없다(레거시 자격증명·네트워크 접근이 사용자 환경에 있음). 020 과 동일하게 **옵션 A(사용자 실행 + 결과 전달)** 를 채택한다 — 옵션 B(사용자 전체 리허설 직접 구축·실행)·옵션 C(정적 검증만으로 마감, 실 바이너리 이관이라는 spec 본질상 채택 불가)는 채택하지 않는다.

### 사후 운영 검증 피드백 사이클

본 spec 파이프라인 종료(이관 스크립트·검증 산출물 확정) 후에도, 실제 이관 실행은 사용자 환경(레거시 AWS 접근)에서 이루어지므로 아래 사후 발견 가능 시나리오를 대비한다.

1. 파이프라인 종료 후 사용자가 실제 환경에서 점검할 가능성이 있는 시나리오: (a) 실측 파일 총량이 예상보다 커서 사전 복사 소요가 길어짐(단, NFR-002 로 시간 제약 자체는 없음), (b) 레거시 S3 버킷 구조·키 네이밍이 예상([TO-VERIFY])과 달라 key 그대로 사용 전략(FR-004)이 재검토 필요, (c) 델타 구간에서 예상보다 많은 신규 업로드가 발생해 NFR-001(60분) 위협, (d) 체크섬 비교 방식(ETag 등)이 레거시 스토리지 제공자에 따라 상이.
2. 위 시나리오로 결함이 발견되면: 결함 내용을 본 spec.md "미결 사항" 또는 별도 patch spec 입력으로 사용 → main session 의 "spec 수정" 이벤트 → 1단계 재진입(cycle N+1) 또는 별도 patch spec 진입.
3. 사후 검증 결과는 이관 실행 완료 후 `docs/specs/v1.1.0/CHANGES.md` 및 `context.md` 갱신을 통해 추적한다.

## 미결 사항

없음 — 사용자가 전 질문(Q1~Q7-2)에 대해 제시된 권장안을 그대로 채택("권장으로 구현", Discord, 2026-07-06)함으로써 모든 핵심 불확실성(이관 대상 범위·이관 시점·이관 방식·key/url 처리·레거시 접근 방식·규모 및 시간 제약·실행 인프라·실패 처리·검증 기준)이 해소되었다. 잔존하는 레거시 실 구조([TO-VERIFY] — 버킷명·키 네이밍·체크섬 방식 등)는 020 과 동일한 옵션 A 원칙(FR-008)으로 처리되며 spec 진행을 차단하지 않는다.
